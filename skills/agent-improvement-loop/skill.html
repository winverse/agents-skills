agent-improvement-loop

남은 토큰 소진형 실행 전 예/아니오를 묻고, 답에 따라 safe backlog batch 또는 staged ceiling review를 수행하는 루프 스킬.

사용 판단 매트릭스

사용

남은 토큰 활용 여부를 정할 때

먼저 예/아니오로 확인한 뒤 spend-down 또는 ceiling mode로 문서, 테스트, validator, hook, eval, skill trigger를 개선한다.

분기

저장소 성격이 다를 때

skills repo track과 general repo track 중 현재 파일 구조에 맞춰 분기한다.

확인 필요

정책 변경이 필요할 때

skill split, global install, broad rewrite, 삭제성 정리는 사용자에게 묻는다.

건너뜀

확인 없는 자동 소진일 때

사용자 동의 없는 대량 리팩터링, 외부 OSS issue 양산, chat-only 조언은 피한다.

루프 실행 흐름

예/아니오 판단

질문

남은 토큰을 최대한 사용할까요?

남은 토큰을 최대한 사용해서 안전한 backlog를 처리할까요? (예/아니오)를 한 번 묻는다.

예

토큰 소진 모드

안전한 backlog를 작은 multi-agent batch로 처리한다. 문서 동기화, validator/eval 보강, smoke check처럼 되돌리기 쉬운 작업만 고른다.

아니오

상한 유지 모드

토큰을 목표치가 아니라 상한선으로 두고, 한 lane씩 검토한 뒤 다음 lane의 가치가 낮으면 멈춘다.

입출력 스키마

입력

  • 남는 토큰 사용 요청
  • 스킬 호출성 개선 요청
  • repo 품질 개선 요청
  • 문서/검증 불일치 의심

읽기

  • AGENTS / README / docs
  • skills / snippets / history
  • tests / CI / hooks
  • source files and manifests

출력

  • 작은 patch 또는 개선 리포트
  • validator, test, hook, eval
  • 문서 정합성 수정
  • 검증 결과와 open question

검토 경로 차트

Spend-down mode는 예라고 답했을 때만 쓰는 예외다. 기본은 staged review이며, main agent가 subagent 증거를 합친 뒤 다음 lane 진행 여부를 결정한다.

금지와 허용

허용
  • 현재 repo evidence에서 시작한다.
  • chat-only 조언을 파일, 테스트, validator로 바꾼다.
  • 작고 검증 가능한 변경을 선호한다.
  • 예라고 답했을 때만 안전한 소진형 backlog를 batch 처리한다.
  • 불명확한 source of truth는 질문한다.
금지
  • 남는 토큰을 이유로 대량 리팩터링하지 않는다.
  • 예/아니오 확인 없이 spend-down을 시작하지 않는다.
  • 전역 설치나 hook 정책을 몰래 바꾸지 않는다.
  • skill repo 규칙을 일반 app repo에 강요하지 않는다.
  • 검증 없이 문서만 좋아 보이게 만들지 않는다.

프로젝트 연결 예시
