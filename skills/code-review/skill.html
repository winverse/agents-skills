사용 판단

사용

리뷰, PR, diff

코드가 이미 있고 사용자가 review, audit, regression risk, missing tests 검토를 원할 때.

집중

findings 우선

요약보다 correctness, 보안, 데이터 손실, 사용자-visible 회귀를 먼저 판단한다.

결합

증거가 더 필요

브라우저 런타임은 browser-qa, UI 비평은 design-review 같은 별도 검증과 함께 쓴다.

건너뜀

구현 요청

사용자가 fix를 명시하지 않았다면 리뷰 중 파일을 수정하지 않는다.

리뷰 흐름: 위험을 찾고 근거를 붙인 뒤 짧게 끝낸다

핵심 동작 검토 테스트 누락 판단 심각 위험 우선 경로

입출력 스키마

사용자 입력

  • PR, diff, 파일 경로, 구현 설명
  • 요구 동작과 검증 결과
  • 리뷰 범위 또는 우선순위
→

읽는 근거

  • SKILL.md 리뷰 stance
  • review-checklist.md
  • js-ts-style.md
→

응답 출력

  • Findings가 맨 앞
  • [Severity] file:line
  • Open Questions, Summary, Tests

위험 신호 지도

높음

사용자-visible 실패

  • 데이터 손실 또는 stale state
  • auth bypass, XSS, SSRF, secret leak
  • 동시성, 취소, 순서 보장 누락
  • MCP, agent tool, CLI wrapper의 destructive side effect
중간

설계 경계 흐림

  • UI에 domain/persistence 로직 누수
  • flag-heavy 함수와 책임 혼합
  • 공유 helper의 speculative extraction
  • tool output, web content, generated file을 신뢰한 경계 누락
상황 의존

JS/TS 표현

  • 단순 collection 변환의 반복 for/if
  • any와 무리한 type assertion
  • 명확한 loop를 reduce로 숨기는 경우

프로젝트 연결 조각

프로젝트 instruction 파일에는 이 스킬의 SKILL.md를 링크하고, 리뷰 요청에서 구현 수정을 하지 않는다는 경계를 유지한다.

Use $code-review at skills/code-review/SKILL.md when asked for code review, PR review, diff review, implementation audit, regression risk review, missing test review, maintainability review, or JS/TS style review.

금지와 허용: 오용 방지

해야 함
  • findings를 맨 앞에 두고 심각도 순으로 정렬한다.
  • 가능하면 모든 finding에 file:line, 영향, 수정 방향을 붙인다.
  • 테스트 누락은 실제 회귀 가능성과 연결될 때 finding으로 쓴다.
  • collection 변환은 map, filter, flatMap, some, every, find가 더 명확한지 본다.
하지 않음
  • 칭찬, 배경 설명, 긴 요약으로 findings를 묻지 않는다.
  • 사용자가 명시하지 않은 코드 수정을 리뷰 중 수행하지 않는다.
  • 루프가 더 명확한 async sequencing, parser, hot path에 functional style을 강요하지 않는다.
  • SOLID를 의식주처럼 적용하지 않고 실제 책임·의존성 문제에만 쓴다.
