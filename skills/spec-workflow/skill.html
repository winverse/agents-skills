spec-workflow

Workflow suite의 implementation 절반입니다. 이미 PRD, issue, acceptance criteria, ADR, design.md, workflow-state cache가 있는 spec을 TDD, 구현, review, QA, docs sync, improvement seed까지 반복해서 끝내는 실행 loop입니다. 대상 코드 repo에서는 RED evidence를 강제하고, 이 shared skills repo에는 TDD hook을 설치하지 않습니다.

사용 판단 매트릭스

사용

spec이 이미 있음

issue, bug repro, acceptance criteria, selected design direction이 있고 검증 가능한 코드 변경으로 닫아야 할 때.

함께 사용

리뷰와 QA가 필요함

UI는 design-review와 browser-qa, 문서 drift는 sync-docs, commit handoff는 atomic-committer.

먼저 확인

권한과 도구 위험

MCP, external API, network, file write, untrusted content가 있으면 Agent Tool And Security Risk Gate를 확인한다.

hook은 대상 repo

TDD gate 문서 계약

이 repo에서는 hook을 실행하지 않는다. production code가 있는 target repo에서만 project-local hook으로 RED evidence를 강제한다.

겹침 차단

동시 작업 claim

work-claims.md가 있으면 production edit 전에 claimed write set, read-only paths, shared/hotspot owner를 확인한다.

넘김

raw project idea

domain, product, architecture, project structure, design system이 아직 정리되지 않았으면 project-workflow로 되돌린다.

실행 흐름: spec을 작은 검증 loop로 닫는다

기본 진행 반복 또는 보류 판단

동시 작업 gate: production edit 전에 claim을 대조한다

primitive 선택 우선순위: 전체 패키지가 아니라 필요한 역할만 고른다

출력에는 항상 <source package>: <exact name> -> selected | skipped | fallback 형식의 provenance ledger(출처 기록)를 남긴다.

입출력 스키마

파일 관계 지도

skills/spec-workflow/ ├── SKILL.md ├── skill.html ├── references/ │ ├── spec-workflow-playbook.md │ └── upstream-dependency-map.md └── scripts/ └── validate-spec-workflow.ts runtime artifact target: .scratch/<feature-slug>/ ├── spec-notes.md ├── implementation-plan.md ├── test-evidence.md ├── tdd-exception.md ├── qa-evidence.md ├── review-notes.md ├── docs-sync.md ├── work-claims.md ├── workflow-state.md ├── workflow-log.md └── eval-candidates/
workflow-state.md는 반복 질문을 줄이는 cache이고, work-claims.md는 같은 파일을 동시에 수정하지 않게 하는 coordination artifact입니다. authority는 CONTEXT.md, ADR, PRD, issue, design.md에 둡니다.

프로젝트 연결 조각

대상 프로젝트의 AGENTS.md 또는 동등한 instruction file에 링크로 연결한다.

Use $spec-workflow at <skills-repo>/skills/spec-workflow/SKILL.md when a scoped spec, issue, bug, or vertical slice already has PRD, acceptance criteria, ADR, or design.md context and should be implemented through plan review, TDD, implementation, review, QA, document sync, and completion handoff.

검증 게이트

완료 조건
  • acceptance criteria 충족
  • tests 또는 runtime evidence 있음
  • review finding 처리 또는 defer 명시
문서 동기화
  • docs drift 필요 여부 확인
  • workflow log와 artifact hygiene 확인
  • gate 누락이 있으면 eval seed 후보 기록
  • commit/push는 요청 시 handoff만
repo 검증
  • node scripts/validate-skill.ts skills/spec-workflow
  • node scripts/validate-skill-html.ts .
  • 필요 시 browser viewport 확인

금지와 허용

한다
  • local instructions와 spec authority를 먼저 읽는다.
  • primitive마다 출처와 selected | skipped | fallback 상태를 표시한다.
  • work-claims.md가 있으면 write set 겹침을 먼저 확인한다.
  • 가능하면 production change 전에 RED evidence를 남긴다.
  • target code repo에서는 RED, characterization, TDD exception 중 하나 없이는 production edit을 통과시키지 않는다.
  • 큰 spec에서만 disjoint write set으로 subagent fan-out을 쓴다.
하지 않는다
  • raw idea를 이 skill로 바로 구현하지 않는다.
  • 외부 plugin 내용을 통째로 복사하지 않는다.
  • 다른 active lane의 claimed write set과 겹치면 patch를 만들지 않는다.
  • 바로 다음 작업을 막는 일을 subagent에게 넘기지 않는다.
  • 사용자 요청 없이 commit, push, ship을 실행하지 않는다.
  • 이 shared skills repo에 TDD hook을 설치하거나 실행하지 않는다.
