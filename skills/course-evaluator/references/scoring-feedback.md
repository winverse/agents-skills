# 채점과 피드백 기준

이 reference는 `course-evaluator`가 실제 학생 답안을 평가할 때 읽는다. 루브릭, 학생 답안, 사용자 선호가 충돌하면 사용자 선호를 우선하되, 점수 산정 근거는 명확히 남긴다.

## 입력 파싱 기준

- CSV는 RFC 4180 호환 파서나 런타임의 구조화 CSV 기능을 사용한다.
- 답안 cell에는 줄바꿈, 쉼표, 따옴표가 들어갈 수 있다고 가정한다.
- 학생 이름, 문항 column, 제출 여부, 총점 column을 먼저 식별한다.
- 파싱 후 학생 수, 문항 수, 빈 답안 수를 검산한다.
- line-based parsing으로 답안을 잃었을 가능성이 있으면 중단하고 다시 파싱한다.

## 채점 정책 기준

| 사용자 표현 | 적용 방식 |
| --- | --- |
| `러프하게` | 개념 방향이 맞으면 관대하게 점수를 준다. 단, 빠진 핵심은 코멘트로 설명한다. |
| `고득점을 위해서` | 단순 감점보다 고득점 답안으로 확장하는 방법을 알려준다. |
| `거의 모범 정답까지` | 학생 답안의 부족한 부분을 모범 답안 수준의 개념 문장으로 보완한다. |
| `같은 스타일` | 이전 평가의 말투와 파일 구조를 유지하되, 코멘트 문장은 학생별로 새로 쓴다. |
| `합쇼체` | 문장 끝을 정중하고 단정한 한국어로 맞춘다. |

## 코멘트 구조

문항별 코멘트는 다음 순서를 따른다.

1. 학생이 맞힌 내용을 먼저 인정한다.
2. 학생 답안의 표현을 근거로 이해 수준을 짚는다.
3. 빠진 개념이나 오해를 구체적으로 말한다.
4. 고득점을 위해 추가해야 할 개념 문장을 제시한다.

좋은 코멘트는 루브릭 문장을 복사하지 않는다. 같은 감점 사유라도 학생이 무엇을 썼고 무엇을 쓰지 않았는지가 드러나야 한다.

## 반복 방지 점검

- 여러 학생 파일에서 같은 문장 시작이 3번 이상 반복되면 재검토한다.
- `전반적으로`, `좋습니다`, `조금 더` 같은 일반 문구만으로 끝나는 코멘트를 피한다.
- 코멘트가 학생 이름 없이도 누구 답안인지 추론 가능해야 한다.
- 빈 답안은 빈 답안이라고 쓰되, 해당 문항의 핵심 개념을 짧게 제시한다.

## 출력 파일 기준

기본 파일 구조는 다음처럼 둔다.

```text
평가/
└── 학생명.md
```

파일명 충돌, 특수문자, 동명이인이 있으면 안전한 suffix를 붙인다. 원본 CSV 전체를 학생 파일에 붙이지 말고, 평가에 필요한 짧은 답안 근거만 인용하거나 요약한다.

## 검증 보고 기준

최종 보고에는 다음을 포함한다.

- 입력 파일과 출력 폴더
- 학생 수와 생성 파일 수
- 적용한 채점 정책
- 반복 코멘트 점검 여부
- 실행한 검증 명령 또는 수동 검산 결과
