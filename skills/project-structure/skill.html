선택 메뉴

긴 구조 질문지 대신 tree가 바뀌는 선택지만 숫자로 묻는다.

프로젝트 종류 1 프론트엔드 web, 2 백엔드 API, 3 풀스택 monorepo, 4 데스크톱 app. 단독 web/API도 기본은 apps/web, apps/api.
API 계약 1 GraphQL 기본, 2 REST, 3 hybrid. REST를 고른 경우에만 controller 경계를 만든다.
DB 1 PostgreSQL + Drizzle 기본, 2 MongoDB + Atlas, 3 Supabase Postgres, 4 DB 없음, 5 외부 DB. 선택한 DB만 tree에 넣는다.
Postgres hosting 1 app-managed Postgres 기본, 2 Supabase managed Postgres, 3 기존 Postgres. Supabase도 기본은 PostgreSQL/psql 호환 DB로 다룬다.
Mongo hosting 1 MongoDB Atlas 기본, 2 self-hosted MongoDB, 3 기존 MongoDB. Atlas network와 required secret names를 문서화한다.
DB shell 1 PostgreSQL/Supabase는 psql, 2 MongoDB는 mongosh, 3 shell script 없음. helper는 검증된 env를 사용한다.
캐시 1 필요 시 cache boundary로 Redis 사용, 2 Redis 없음. monorepo Redis는 packages/db/src/redis.
인증 1 필요 시 app 소유 auth module, 2 외부 auth, 3 auth 없음. security folder를 선택에 맞춘다.
생성 산출물 1 ignored and reproducible 기본, 2 committed and reviewed. GraphQL/Panda 산출물 정책을 하나로 고른다.
데스크톱 shell 1 Tauri + Vite React 기본, 2 Tauri + 기존 frontend shell. GraphQL 사용 시 desktop autogen 경로를 맞춘다.
인프라 1 요청 시 Pulumi AWS 기본, 2 infra 없음, 3 기존 infra tool. 배포/cloud/Docker 요청이 있을 때만 infra 폴더를 포함한다.
컨테이너 registry 1 Docker image를 AWS ECR로 push 기본, 2 registry 없음, 3 외부 registry. app Dockerfile은 각 app 옆에 둔다.
런타임 target 1 신규 AWS container runtime은 ECS Fargate 기본, 2 VM 제어가 필요하면 EC2 Docker host, 3 cloud runtime 없음.
공개 진입점 1 public AWS service는 ALB 기본, 2 internal service, 3 기존 DNS/CDN. Route 53/ACM은 custom domain/TLS 요청 때만 포함한다.

스택 원칙

기본값은 빠르게 구조를 정하기 위한 출발점이다. 프로젝트 종류가 정해지면 관련 stack만 묻고, 나머지는 기본 정책으로 둔다. GraphQL codegen, Drizzle migration, psql/mongosh helper, Tauri build, deploy script는 env/helper와 선택된 DB/infra 경계를 반드시 거친다.

DB 옵션

기본은 PostgreSQL + Drizzle이다. MongoDB, Atlas, Supabase, psql, mongosh는 선택됐을 때만 tree와 env 정책에 포함한다.

PostgreSQL / Supabase Postgres packages/db/drizzle, packages/db/src/postgres, packages/db/scripts/psql.ts를 둔다. runtime과 migration connection이 다르면 DATABASE_DIRECT_URL을 server-only로 분리한다.
MongoDB Atlas packages/db/src/mongo에 client, collections, indexes를 둔다. MONGODB_URI와 Atlas network access, private endpoint/IP allowlist, DB user를 문서화한다.
Supabase service role SUPABASE_SERVICE_ROLE_KEY는 서버 전용 secret name으로만 문서화한다. frontend public env나 예제 값에 live secret을 넣지 않는다.
shell helper PostgreSQL/Supabase는 psql, MongoDB는 mongosh helper를 둔다. 두 helper 모두 app-local Zod env를 통과한 connection string만 사용한다.
migration 구분 MongoDB에는 Drizzle migration을 만들지 않는다. MongoDB는 seed와 index sync script를 쓰고, PostgreSQL/Supabase만 Drizzle migration owner를 가진다.

env 계약

앱별 schema apps/web, apps/api, apps/desktop가 각자 src/config/env.ts Zod schema를 가진다.
helper 전용 package packages/config는 env mode, dotenv loading, Zod helpers만 제공한다. app 변수 정의는 하지 않는다.
shared env 직접 읽기 금지 공유 패키지는 process.env를 직접 읽지 않는다. 필요한 값은 app에서 주입한다.
프론트엔드 분리 browser 공개 값과 server-only 값을 분리한다. 공개 값만 framework public prefix를 사용한다.
도구도 helper 사용 codegen, migration, Tauri build script가 각자 env를 다시 파싱하지 않고 공용 helper를 호출한다.
infra config 분리 app env schema는 application 변수를 소유하고, Pulumi stack config는 AWS region, account, image tag, domain, capacity, runtime target을 소유한다.
앱별 env 파일 env/.env.development, .env.test, .env.stage, .env.production을 app마다 둔다.
단독 앱도 같은 root standalone frontend/backend라도 기존 repo가 명확히 다르지 않으면 apps/web, apps/api를 써서 나중의 monorepo 전환 비용을 줄인다.

GraphQL codegen 계약

앱 소유 config GraphQL을 쓰는 각 app은 app root에 codegen.ts를 둔다.
schema 소유자 API schema source/output은 apps/api/src/graphql 아래에 둔다.
document 소유자 query/mutation 문서는 src/features/<domain>/graphql에 둔다.
자동 생성 경로 web, API, desktop 모두 <app>/src/graphql/autogen.ts를 entrypoint로 쓴다.
산출물 정책 generated 파일은 ignored/reproducible 또는 committed/reviewed 중 하나로 통일한다.
Turbo 작업 monorepo에서는 codegen을 app scripts와 turbo task에 연결한다.

인프라 정책

deployment, cloud, Docker, Pulumi, AWS, ECR, ECS, EC2, CI/CD 배포 구조가 요청된 경우에만 추가한다.

Pulumi 소유 infra/pulumi가 network, IAM, ECR, ECS/EC2, load balancer, security group, logs, runtime wiring을 소유한다.
Docker 위치 <app>/Dockerfile과 <app>/.dockerignore를 app 옆에 둔다. local Compose는 production runtime으로 보지 않는다.
ECR image CI/CD는 Docker image를 build하고 immutable tag를 AWS ECR에 push한 뒤, 선택한 image tag로 pulumi up을 실행한다.
런타임 선택 새 AWS container runtime은 ECS Fargate가 기본이다. SSH, host customization, 비컨테이너 host process가 필요할 때 EC2를 고른다.
진입점과 TLS public service는 ALB를 기본 진입점으로 둔다. Route 53과 ACM은 custom domain이나 TLS가 선택된 경우에만 tree에 넣는다.
비밀값 금지 Pulumi stack file에 live secret을 쓰지 않는다. secret provider 또는 placeholder와 필요한 secret name만 문서화한다.
배포 확인 선택한 health endpoint로 post-deploy smoke check를 실행하고, immutable image tag와 Pulumi stack update를 배포 단위로 기록한다.

구조 검증

env 일치 모든 app은 env/*, src/config/env.ts, Zod schema를 같은 상대 경로로 가진다.
app root 일치 단독 web/API 요청도 기본은 apps/web 또는 apps/api이며, repo-root layout은 사용자나 기존 구조가 분명할 때만 쓴다.
DB 경계 PostgreSQL/Supabase는 packages/db/drizzle과 postgres/, MongoDB Atlas는 mongo/와 index sync, Redis는 packages/db/src/redis에 둔다.
API 준비 상태 logger, request logging, health/readiness, security plugins, observability 경계를 확인한다.
테스트 범위 unit/component는 colocated, e2e는 app-level test/e2e 또는 API test에 둔다.
인프라 범위 infra-aware repo는 Pulumi AWS, app-local Dockerfile/.dockerignore, ECR image owner, public entrypoint, required secret names, smoke check, ECS Fargate 또는 EC2 중 정확히 하나의 runtime path를 표시한다.

금지와 허용

허용

  • 먼저 숫자로 프로젝트 종류를 확정한다.
  • route는 얇게, domain은 features에 둔다.
  • NestJS는 modules/common/providers 경계를 유지한다.
  • 중복 folder 역할을 최소화하고 app/domain/shared 경계를 명확히 한다.
  • 최종 tree가 선택한 folder 규칙을 지켰는지 검증한다.
  • API logger는 provider에, Redis DB boundary는 monorepo의 packages/db에 둔다.
  • env helper를 codegen, migration, build에도 재사용한다.
  • GraphQL, Panda, test, health, observability 위치를 tree에 표시한다.
  • infra 요청이 있으면 Pulumi AWS, Docker, ECR, ECS/EC2 runtime 경계를 tree에 표시한다.
  • MongoDB Atlas와 Supabase Postgres를 선택한 경우 DB provider, shell helper, server-only secret 경계를 표시한다.

금지

  • 기본값으로 full-stack single repo를 제안하지 않는다.
  • shared package에서 process.env를 읽지 않는다.
  • GraphQL 기본 프로젝트에 REST controller를 기본 생성하지 않는다.
  • 배포 요청이 없는데 infra/, Docker, Pulumi 구조를 기본 생성하지 않는다.
  • Pulumi stack file에 live secret을 쓰지 않는다.
  • 기존 repo를 조사하지 않고 대량 이동하지 않는다.
  • migration 파일을 drizzle/와 src/migrations/에 중복해서 두지 않는다.
  • MongoDB 선택 시 Drizzle migration을 만들거나 Supabase service-role key를 frontend public env에 노출하지 않는다.
