project-structure

frontend, backend, full-stack monorepo, desktop app, infrastructure-aware 구조를 선택하고 folder boundary, env, codegen, DB, test, security, health/readiness, observability, infra handoff까지 한 번에 정리하는 스킬.

사용 판단

사용

구조 선택이 목적

web, API, monorepo, desktop, infra-aware repo의 folder tree와 경계가 필요할 때 호출한다.

질문

stack이 불명확

숫자 선택(compact numeric choices)으로 project kind, runtime, DB, infra 여부처럼 tree가 바뀌는 항목만 묻는다.

연결

workflow 이후

raw product discovery는 먼저 project-workflow가 처리한다. 이 스킬은 구조 질문이 구체화된 뒤 사용한다.

건너뜀

infra 요청 없음

사용자가 cloud, deploy, Docker, Pulumi, AWS를 요구하지 않았다면 infra folder를 추가하지 않는다.

실행 흐름

순서 있는 workflow 완료: 선택한 구조, folder tree, 경계, 검증이 함께 있음

기본 stack 우선순위

MongoDB/document DB가 선택되면 MongoDB + Atlas, managed psql-compatible Postgres가 선택되면 Supabase Postgres 경계를 사용한다.

검증 게이트

HTML 변경 검증
  • node scripts/validate-skill.ts skills/project-structure

  • node scripts/validate-skill-html.ts .
구조 답변 검증
  • 사용자가 선택한 project kind에 없는 app type을 tree에 넣지 않는다.
  • security, tool, migration, deploy, secret, scrubbed artifact boundary를 표시한다.
  • 최종 답변은 선택한 구조, 폴더 트리, 경계, 검증 순서를 따른다.

금지와 허용

해야 함
  • 기존 repo가 있으면 구조와 package manager를 먼저 읽는다.
  • tree가 바뀌는 선택만 compact numeric choices로 묻는다.
  • meaningful boundary folder에 짧은 AGENTS.md index를 둔다.
  • health/readiness와 observability 위치를 app 또는 provider boundary에 포함한다.
하지 않음
  • raw product discovery 단계에서 이 스킬만으로 PRD나 domain language를 만든다.
  • 사용자가 infra를 요청하지 않았는데 Pulumi, Docker, AWS 구조를 넣는다.
  • Supabase service-role key나 MongoDB URI를 client-visible env에 둔다.
  • source artifact와 generated artifact를 같은 owner 없이 섞는다.
