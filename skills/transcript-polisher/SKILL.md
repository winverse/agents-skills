# Transcript Polisher 전사본 다듬기

## 핵심 규칙

텍스트 다듬기를 치환표, 정규식, 일괄 스크립트, 코드 기반 문자열 치환으로 해결하지 않는다. 에이전트가 텍스트를 직접 읽고, 앞뒤 문맥과 강의 흐름을 이해한 뒤 판단으로 수정한다.

스크립트나 셸 명령은 파일 찾기, 줄 번호 확인, 섹션 개수 확인, 구조 검증에만 사용한다. 본문 수정의 엔진으로 사용하지 않는다.

## 작업 절차

1. 대상 텍스트 파일을 확인하고 사용자의 요청에서 source root, output root, output mode, 출력 밀도, 완료 목표를 결정한다.
   - 원본 직접 수정: 기존 파일을 수정한다.
   - 병렬 출력: 요청된 출력 폴더 아래에 같은 상대 경로로 수정본을 만든다.
   - 출력 밀도 기본값: 사용자가 요약이나 강의 노트를 명시하지 않으면 `polish`로 처리한다.
   - 목표 루프: Claude Code의 `/goal` 또는 이에 준하는 장기 작업이면 완료 조건, 증명 방법, 제약, 최대 턴 또는 시간 조건을 먼저 적는다.
2. 각 파일을 충분히 읽어 강의, 문서, 이야기의 전체 흐름을 파악한다.
3. 반복 등장하는 고유명사와 기술 용어를 먼저 작은 용어 검토 장부로 잡는다.
   - 장부는 자동 치환표가 아니다. 직접 읽고 판단할 때 일관성을 확인하는 체크리스트로만 쓴다.
   - 같은 STT 오류가 여러 표기로 흔들리면 주변 문맥을 읽고 표준 표기를 정한다.
   - 검색 명령으로 후보 위치를 찾을 수는 있지만, 각 수정은 해당 문맥을 확인한 뒤 직접 적용한다.
   - 용어 장부는 `확정 용어`와 `불확실 용어`를 분리한다.
   - 오디오나 외부 근거 없이는 확정할 수 없는 제품명, 사람 이름, 프로젝트명은 본문에서 억지로 표준화하지 않는다.
   - 불확실 용어는 최종 보고에 원문 표기, 위치, 왜 확정하지 않았는지 함께 남긴다.
4. 대상을 의미 있는 편집 단위로 나눈다.
   - 제목, 빈 줄 문단, 장면 전환, 강의 주제 전환을 우선 기준으로 삼는다.
   - 뚜렷한 제목이 없으면 인접 문단을 묶어 독립 검토 가능한 단위로 만든다.
   - 한 문장씩 너무 작게 나누지 않는다. 문맥이 유지될 만큼 크게 나눈다.
5. 검토 작업을 맡기기 전에 줄 번호나 안정적인 섹션 라벨을 확보한다.
6. 현재 런타임에 명시적인 서브 에이전트 또는 subagent delegation 도구가 있고, 사용자가 요청했거나 활성 런타임 정책이 자동 위임을 허용할 때만 편집 단위마다 서브 에이전트를 하나씩 실행한다.
7. 서브 에이전트 도구가 없으면 위임했다고 말하지 않는다. 제약을 밝히고 같은 섹션 계획을 메인 에이전트가 직접 수행한다.
8. 각 서브 에이전트에는 담당 단위와 최소한의 앞뒤 문맥만 제공한다. 전체 파일을 불필요하게 넘기지 않는다.
9. 서브 에이전트에는 장황한 감상이 아니라 담당 단위의 수정안, 불확실한 표현, 주변 섹션과 맞춰야 할 용어를 요청한다.
10. 메인 에이전트가 모든 결과를 다시 통합한다.
   - 한 문서 안의 말투와 용어를 일관되게 맞춘다.
   - 인접 섹션 사이의 충돌이나 반복을 정리한다.
   - 사용자가 재작성하라고 하지 않았다면 원래 의미를 보존한다.
   - 오디오 확인 없이는 확정할 수 없는 표현은 임의로 꾸며내지 않고 불확실하다고 남긴다.
11. 수정은 텍스트를 읽은 뒤 `apply_patch` 또는 이에 준하는 직접 편집 도구로 적용한다. 스크립트가 새 본문을 자동 생성하게 하지 않는다.
12. 목표 루프를 돌린다. 완료 증거가 부족하면 다음 편집 단위나 약한 검증 지점으로 돌아간다.
13. 수정 후 파일을 처음부터 다시 읽어 흐름, 중복 문단, 끊긴 전환, 남은 어색한 용어를 확인한다.

## 입력/출력 루트 결정

긴 전사본 묶음을 처리할 때는 먼저 source root와 output root를 결정한다. 특정 repo의 디렉터리 이름을 스킬 규칙으로 고정하지 않고, 사용자의 명시값과 현재 폴더 구조를 보고 결정한다.

1. 사용자가 source root와 output root를 명시하면 그대로 따른다.
2. 사용자가 명시하지 않았고 현재 작업 폴더에 `transcripts_raw/`가 존재하면 source root 기본값으로 제안하거나 사용한다.
3. 사용자가 명시하지 않았고 검토본 출력이 필요하면 output root 기본값으로 `transcripts_reviewed/`를 제안하거나 사용한다.
4. source/output 후보가 여러 개이거나 기본값을 확신할 수 없으면 작업 전에 짧게 확인한다.
5. output root의 기존 내용을 삭제하거나 덮어쓰는 작업은 사용자가 명시적으로 요청했을 때만 수행한다.
6. 병렬 출력 모드에서는 source root 아래 상대 경로를 output root 아래에 그대로 보존한다.
7. output을 재생성하거나 이전 수정 패치를 재적용할 때는 적용 대상이 source root가 아니라 output root인지 먼저 확인한다.
8. 실수로 source root에 수정이 들어간 정황이 있으면 즉시 되돌리고, source root에 raw 특유의 오류 후보가 그대로 남아 있는지 확인한다.
9. 완료 보고에는 source root, output root, source 파일 수, output 파일 수, 상대 경로 일치 여부를 포함한다.

## 출력 밀도 결정

전사본을 다듬을 때는 작업 시작 전에 output mode를 정한다. 사용자가 명시하지 않으면 `polish`를 기본값으로 둔다.

| Mode | 목적 | 분량 기준 | 사용 시점 |
| --- | --- | --- | --- |
| `polish` | 원문 전사본을 자연스럽게 다듬는다. | 원문 흐름과 대부분의 정보량을 유지한다. 불필요한 반복 파편만 줄인다. | `reviewed`, `다듬어줘`, `전사본 수정`, 별도 요약 요청이 없는 경우 |
| `lecture-notes` | 강의 노트처럼 읽기 좋게 압축한다. | 중복과 구어 반복을 적극적으로 줄인다. | 사용자가 노트화, 정리본, 압축을 요청한 경우 |
| `summary` | 핵심만 요약한다. | 원문 구조보다 핵심 주제와 결론을 우선한다. | 사용자가 요약, 핵심만, 짧게를 요청한 경우 |

`polish` 모드에서는 source 파일 대비 output이 지나치게 짧아지면 실패 신호로 본다. 정확한 비율은 문서 성격에 따라 달라질 수 있지만, 원문 대비 절반 이하로 줄었다면 요약이나 강의 노트로 흐른 것인지 재검토한다. 사용자가 명시적으로 압축을 원하지 않았다면 다시 원문 흐름을 보존하는 방향으로 수정한다.

반대로 `polish` 모드에서 source를 그대로 복사하고 완료라고 말하는 것도 실패다. raw 보존 baseline을 만들 수는 있지만, 그 뒤에는 직접 읽으면서 명백한 STT 오류, 깨진 조사, 끊긴 문장, 용어 불일치를 고쳐야 한다. 분량 보존은 원문을 그대로 두라는 뜻이 아니라, 원문 정보량과 강의 흐름을 유지하면서 문장을 읽기 좋게 다듬으라는 뜻이다.

`lecture-notes`나 `summary` 모드에서는 압축이 허용되지만, 이 경우 최종 보고에 해당 모드를 명시한다.

## Claude Code `/goal` 방식 목표 루프

이 스킬은 Claude Code `/goal` 명령 자체를 실행하는 스킬이 아니다. 대신 Claude Code `/goal`의 작업 방식, 즉 “완료 조건을 정하고, 각 턴 뒤 별도 평가자가 대화에 드러난 증거로 조건 충족 여부를 판단하며, 충족되지 않으면 다음 턴을 계속하는 루프”를 전사본 편집에 적용한다.

Claude Code 기준에서 중요한 차이는 평가자가 도구를 직접 실행하거나 파일을 새로 읽지 않는다는 점이다. 따라서 완료 조건은 에이전트가 대화와 최종 보고에 증거를 드러낼 수 있는 형태여야 한다.

이 루프는 자가 개선 스킬이라기보다 완료 조건 감사 장치에 가깝다. 약한 부분을 다음 턴에서 계속 고치도록 만들 수는 있지만, 개선 여부는 대화에 드러난 증거와 명시된 검증 결과로 판단된다.

목표 루프가 필요한 경우:

- 여러 파일 또는 긴 전사본을 끝까지 다듬어야 한다.
- 한 번의 검토로 완료 여부를 믿기 어렵다.
- 사용자가 “루프를 돌아서”, “완료될 때까지”, “/goal처럼” 처리하라고 했다.
- 섹션별 서브 에이전트 결과를 다시 합쳐 전체 품질을 검증해야 한다.

루프는 아래 순서로 반복한다.

1. 완료 조건 작성: 어떤 source root, 어떤 output root, 어떤 output mode, 어떤 품질 기준이면 완료인지 한 문장으로 적는다.
2. 증명 방법 포함: “어떻게 증명할 것인가”를 조건 안에 넣는다. 예: source/output 파일 수 일치, 상대 경로 보존, `polish` 모드 분량 보존, 섹션별 검토 완료, 남은 불확실 표현 목록 작성.
3. 제약 포함: 원본 의미 보존, 코드 치환 금지, 생성 예시 보존 같은 절대 조건을 함께 적는다.
4. 범위 제한 포함: 오래 걸릴 수 있으면 “최대 N턴 또는 N분 후 남은 항목 보고”처럼 종료 조건을 둔다.
5. 작업 실행: 가장 약한 항목부터 읽고 수정한다.
6. 평가자 관점 보고: 각 루프 끝에서 Claude Code `/goal` 평가자가 볼 수 있도록 현재 증거를 대화에 남긴다.
7. 계속 또는 완료: 완료 조건을 증명할 대화상 증거가 부족하면 다음 루프를 돈다. 모든 조건이 드러난 증거로 입증될 때만 완료로 보고한다.

Claude Code `/goal`이 실제로 활성화된 환경이라면 한 세션에 하나의 goal만 활성화된다. 새 goal은 기존 goal을 대체할 수 있으므로, 스킬 사용 중에는 목표를 더 작거나 쉬운 조건으로 바꾸지 않는다. 중간에 멈춰야 할 때는 `/goal clear` 또는 사용자의 중단 지시가 필요할 수 있다.

### Claude Code `/goal` 조건 예시

```text
/goal transcripts_reviewed/llm_part1 아래 모든 txt 파일을 직접 읽고, 코드 치환 없이 문단 단위로 어색한 STT 오류와 강의 흐름을 다듬는다. 완료 조건은: 대상 txt 파일 수와 수정/검토 완료 파일 수가 일치하고, 각 파일의 섹션 검토 여부가 보고되며, 불확실해서 오디오 확인이 필요한 표현이 별도 목록으로 남고, 마지막 보고에 검증 증거가 포함되는 것이다. 최대 20턴이 지나면 완료하지 말고 남은 파일과 이유를 보고한다.
```

`polish` 모드 예시는 아래처럼 분량 보존 조건을 포함한다.

```text
/goal source root의 모든 txt 파일을 직접 읽고, output root에 같은 상대 경로로 polish 검토본을 만든다. 원문 의미와 강의 순서를 유지하고, 요약본이나 강의 노트로 압축하지 않는다. 완료 조건은: source/output txt 파일 수와 상대 경로가 일치하고, 각 output 파일이 source 대비 과도하게 짧아지지 않았으며, 의도적 생성 예시는 보존되고, STT 오류와 문장 흐름만 다듬었다는 검증 증거가 최종 보고에 포함되는 것이다. 최대 10회 반복 후에도 개선점이 남으면 남은 이유와 다음 수정 후보를 보고한다.
```

## 반복 개선 루프

사용자가 “개선점이 없을 때까지 반복”, “최대 N번”, “다시 만들고 스킬도 고쳐라”라고 요청하면 작업 루프와 스킬 개선 루프를 함께 관리한다.

1. 각 반복 시작 전에 현재 스킬 규칙으로 작업한다.
2. 반복 중 스킬 규칙의 빈틈이 발견되면 다음 반복 전에 `SKILL.md`에 반영한다.
3. 스킬 개선은 범용 규칙으로만 적는다. 특정 repo의 디렉터리명, 파일명, 강의명은 기본값 예시나 프로젝트 지시로만 둔다.
4. 개선 후보가 단순 작업 실수인지, 스킬 규칙 부재인지 구분한다. 작업 실수는 결과물을 고치고, 반복될 수 있는 규칙 부재는 스킬 문서를 고친다.
5. 최대 반복 횟수에 도달하면 완료라고 말하지 말고 남은 개선점과 중단 이유를 보고한다.
6. 개선점이 없다고 판단하려면 파일 구조, 출력 밀도, 문맥 흐름, 생성 예시 보존, 불확실 표현 보고가 모두 충족되어야 한다.
7. 반복 중 output이 source와 사실상 동일하고 명백한 STT 오류가 남아 있으면 “분량 보존”은 통과해도 `polish` 완료는 실패로 본다.

## 완료 감사 기준

완료라고 말하기 전에 다음 증거를 확보한다.

- 대상 파일 목록과 수정 완료 파일 목록이 일치한다.
- 병렬 출력 모드에서는 source root와 output root의 상대 경로가 일치한다.
- source root가 원본 상태로 보존되었고 수정은 output root에만 적용되었다.
- `polish` 모드에서는 output이 요약본이나 강의 노트로 과도하게 압축되지 않았다.
- `polish` 모드에서는 source를 단순 복사한 상태가 아니라 명백한 STT 오류와 문장 흐름 문제가 수정되었다.
- 각 파일의 편집 단위가 모두 검토되었다.
- 서브 에이전트를 쓴 경우 각 단위의 결과가 메인 에이전트에 의해 통합되었다.
- 스크립트 치환이나 regex로 본문을 자동 수정하지 않았다.
- 전체 파일을 다시 읽었고 강의 흐름이 유지된다.
- 불확실한 오디오 표현은 확정 수정하지 않고 별도 목록으로 남겼다.
- 검토만 한 경우에는 파일/줄 번호가 있는 문제 목록을 남겼다.
- Claude Code `/goal` 평가자가 판단할 수 있도록 위 증거를 최종 보고나 대화에 명시했다.

## 서브 에이전트 지시문

각 섹션에는 아래처럼 좁은 작업을 맡긴다.

```text
이 전사본 섹션을 읽고 어색한 STT 용어, 깨진 한국어, 문단 흐름, 강의 연결을 검토해라.
반환할 것:
- 이 섹션의 수정본
- 오디오나 사람 확인이 필요한 불확실한 표현
- 인접 섹션과 일관되게 유지해야 할 용어
강의를 요약하지 말 것. 원문에 없는 사실을 추가하지 말 것.
```

섹션이 많아도 가능하면 사용자가 요청한 “섹션 수만큼 서브 에이전트” 대응을 유지한다. 런타임 제한, 도구 정책, 안전 규칙 때문에 섹션마다 서브 에이전트를 실행할 수 없으면 그 이유를 명확히 말하고 같은 섹션 계획을 로컬로 수행한다.

## 편집 기준

- 강사의 구어체 설명 톤을 유지한다.
- 기술 용어는 문서 전체에서 일관되게 쓴다.
- 다만 확정 근거가 부족한 고유명사는 일관성만을 이유로 임의 표준화하지 않는다.
- 명백한 STT 오류, 조사 오류, 띄어쓰기, 반복 파편을 고친다.
- 주변 문맥상 중복이 확실할 때만 중복 문장을 제거한다.
- 강의 예시로 등장하는 무의미한 생성어, 어색한 모델 출력, 의도적인 오류는 함부로 자연스럽게 고치지 않는다.
- 원본이 강의체이면 지나치게 문어체로 바꾸지 않는다.
- LLM, n-gram, token, context, model 같은 일반 AI 용어는 필요하면 그대로 둔다.
- 흐름이 깨진 경우에는 단어 하나만 고치기보다 문단 단위로 정리한다.

## 결과 형식

검토만 요청받으면 다음을 보고한다.

- 전체 흐름 판단
- 파일 또는 섹션별 문제를 우선순위 순서로 정리
- 파일 경로와 줄 번호가 있는 예시
- 최종 검토본으로 사용할 수 있는지 여부
- 남은 목표 루프 항목 또는 완료 증거

수정까지 요청받으면 파일을 갱신한 뒤 다음을 보고한다.

- 수정한 파일
- 수정한 섹션
- 적용한 output mode
- source/output 파일 수와 상대 경로 일치 여부
- `polish` 모드에서 분량이 과도하게 압축되지 않았는지 여부
- source와 동일 복사로 남지 않았고 명백한 STT 오류를 직접 수정했는지 여부
- 오디오나 사람 확인이 필요한 남은 표현
- 불확실 용어를 확정하지 않고 남긴 경우 그 위치와 이유
- 수행한 검증
- 목표 루프 체크리스트의 완료 여부

## 검증 체크리스트

마무리 전에 확인한다.

- 수정본이 원래 강의 순서를 유지하는가.
- 원시 출력이나 생성 예시를 잘못 다듬어 의미를 바꾸지 않았는가.
- 섹션 시작과 끝이 갑자기 끊기지 않는가.
- 반복 문장은 우발적 중복일 때만 제거했는가.
- 파일을 처음부터 끝까지 읽었을 때 눈에 띄는 문맥 점프가 없는가.
- output mode가 사용자 요청과 일치하는가.
- `polish` 모드 결과가 요약이나 강의 노트처럼 과도하게 줄어들지 않았는가.
- `polish` 모드 결과가 source 그대로 복사된 상태로 남아 있지 않은가.
