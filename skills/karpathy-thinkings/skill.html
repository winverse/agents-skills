Decision Matrix

코딩 작업에서 언제 이 사고방식을 강하게 적용할지 판단합니다.

Use

구현, 디버깅, 리팩터링, 리뷰처럼 코드 변경의 판단 품질이 중요한 작업.

Intensify

요구사항이 애매하거나 기존 코드와 충돌하거나 검증 기준이 흐릿한 작업.

두 해석이 서로 다른 구현을 만들면 추측하지 말고 짧게 확인합니다.

Skip

문서 동기화, 스킬 업데이트, 커밋처럼 더 구체적인 repo 스킬이 직접 커버하는 경우.

Four Principles

Karpathy식 사고의 네 축을 구현 행동으로 압축했습니다.

1

Think Before Coding

모호함, 가정, tradeoff를 먼저 드러내고 구현이 갈리는 질문은 확인합니다.

2

Simplicity First

가장 작은 boring solution을 고르고 speculative abstraction을 만들지 않습니다.

3

Surgical Changes

요청과 연결된 파일과 줄만 건드리고 주변 개선은 별도 이슈로 남깁니다.

4

Goal-Driven Execution

성공 기준을 정하고 최소 검증 루프를 돌린 뒤 완료를 보고합니다.

Workflow

작게 생각하고, 작게 고치고, 확인 가능한 기준으로 끝냅니다.

## Project Skills

- Use $karpathy-thinkings at <skills-root>/skills/karpathy-thinkings/SKILL.md when coding, reviewing, refactoring, debugging, or planning implementation work that should follow Karpathy-style agent discipline: think before coding, keep solutions simple, make surgical changes, and verify success criteria.

## Project-Specific Overrides

- Do not silently assume ambiguous requirements; ask when the answer changes the implementation.
- Prefer the smallest boring implementation that satisfies the current goal.
- Touch only files and lines connected to the task.
- Define and run the smallest meaningful verification before calling the work done.

Do

  • 구현 전 성공 기준과 가정을 드러냅니다.
  • 요청 범위와 기존 스타일을 우선합니다.
  • 작게 고치고 작게 검증합니다.

Do Not

  • 애매한 요구를 추측으로 확정하지 않습니다.
  • 필요 없는 추상화와 설정을 만들지 않습니다.
  • 주변 코드를 조용히 리팩터링하지 않습니다.
