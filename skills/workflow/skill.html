사용 판단 매트릭스

사용

새 프로젝트

아이디어를 제품 언어, ADR, PRD, vertical issue로 바꿔야 할 때.

사용

새 feature

기존 docs를 읽고 다음 workflow 단계와 산출물을 정해야 할 때.

함께 사용

구조/리서치

stack 구조는 domain language 이후 `project-structure`로 검증하고, 최신 외부 사실은 `web-research`로 확인한다.

건너뜀

작은 코드 수정

이미 issue와 plan이 명확한 단순 버그 수정은 직접 구현 스킬이 더 낫다.

이 repo의 `workflow` 스킬은 위 이름을 숨기지 않는다. 설치되어 있으면 해당 skill을 직접 호출하고, 없으면 같은 역할을 fallback으로 수행한다.

업스트림 추적

업데이트를 따라가는 방법

  • 원본 skill이 바뀌면 먼저 해당 SKILL.md를 다시 읽는다.
  • workflow에는 전체 내용을 복사하지 않고 handoff, 순서, 산출물, validator만 갱신한다.
  • 출처 묶음은 mattpocock/skills, garrytan/gstack, Superpowers local skills로 기록한다.

방향 우선

  • 도메인과 제품 판단을 건너뛰고 빠르게 구현하지 않는다.
  • PRD 전에는 office-hours와 CEO/spec review를 거친다.
  • 큰 UI는 design.md, plan review, mockup 선택 전에는 구현하지 않는다.

자세한 추적 표와 중복 방지 기준은 upstream-dependency-map.md에 둔다.

중복 방지 흐름

제품 검증

office-hours

누가 원하는지, 지금 해결 방식은 무엇인지, 가장 작은 useful slice와 제거할 범위를 정한다.

구현 설계

brainstorming

triage된 issue 하나의 behavior, file boundary, UI state, test boundary를 좁힌다.

UI 선택

mockup 선택

2-3개 화면 방향을 비교하고 사용자가 고른 방향을 `design.md`와 issue/spec에 기록한다.

같은 “브레인스토밍”이라도 제품 검증, 구현 설계, UI 시안 선택은 서로 대체하지 않는다.

PRD 설정

쓰기 전 고정

  • 경로: .scratch/<feature-slug>/PRD.md
  • 입력: CONTEXT.md, docs/adr/, office-hours, plan-ceo-review, 관련 specs/
  • 언어: 한국어 설명, code-facing canonical term은 영어

검증 기준

  • 첫 usable slice가 한 문장으로 보인다.
  • 포함 범위와 제외 범위가 분리된다.
  • stack과 app boundary는 ADR을 참조한다.
  • data source of truth가 명확하다.

to-prd는 자유 요약이 아니라 위 설정을 잠근 뒤 실행한다. 입력 문서가 충돌하면 PRD를 쓰기 전에 사용자에게 묻는다.

품질 게이트

완료 선언은 code가 아니라 PRD/ADR/issue/test/UI evidence/docs가 같은 말을 할 때 가능하다.

검증 명령

node scripts/validate-skill.ts skills/workflow
node skills/workflow/scripts/validate-workflow.ts skills/workflow
node scripts/validate-skill-html.ts .
node scripts/validate-skill-repo.ts .

금지와 허용

허용
  • 다음 workflow step을 하나로 좁힌다.
  • missing source of truth는 질문으로 막는다.
  • 오래된 plan은 보존하고 현재 truth는 docs/log에 남긴다.
금지
  • 아이디어 단계에서 바로 코드부터 작성하지 않는다.
  • DB-only, API-only, UI-only issue를 기본값으로 만들지 않는다.
  • 검증 산출물을 repo root에 흩뿌리지 않는다.
